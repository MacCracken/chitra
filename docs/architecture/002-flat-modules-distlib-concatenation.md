# 002 — Flat domain modules + distlib strip-include concatenation

Architecture notes describe *how the world is* — non-obvious invariants a reader
cannot derive from the code alone. Not decisions (those live in
[`../adr/`](../adr/)) and not guides (those live in
[`../guides/`](../guides/)). Numbered chronologically — never renumber.

## The invariant

chitra ships as a **single concatenated file**, `dist/chitra.cyr`, that every
consumer links directly (there is no CLI — chitra is a library; see
[`../../CLAUDE.md`](../../CLAUDE.md)). That bundle is produced mechanically by
`cyrius distlib` (= `make dist`, [`../../Makefile`](../../Makefile) target
`dist`). For the mechanical concatenation to yield a **compile-clean** file, the
source tree obeys one rule:

> **Stdlib `include` directives appear in exactly one file: `src/lib.cyr`. The
> domain modules — `error.cyr`, `png_chunks.cyr`, `png_filter.cyr`,
> `png_color.cyr`, `png.cyr` — are FLAT: they carry no `include` lines at all.**

`src/lib.cyr` is the single include chain ([`../../src/lib.cyr`](../../src/lib.cyr)):
it pulls the stdlib (`string`, `fmt`, `alloc`, `vec`, `str`, `io`, `args`,
`syscalls`, `flags`, `assert`, `bench`, then `thread`, then `sankoch`) and then
the five domain modules in dependency order. That file is what the smoke build
(`programs/smoke.cyr` → `build/chitra_smoke`, `make build`) compiles to prove the
whole chain parses; it is **not** itself part of the distributed bundle.

A `grep -n include src/*.cyr` shows every `include` living in `lib.cyr` and none
in the five domain modules. That is the invariant, made visible.

## How `cyrius distlib` uses it

`distlib` reads `[lib].modules` from
[`../../cyrius.cyml`](../../cyrius.cyml), then **strip-concatenates** those
modules — in the declared order — into `dist/chitra.cyr`. "Strip" means any
`include` lines are removed during concatenation: an embedded
`include "lib/alloc.cyr"` inside a bundled file would otherwise survive into the
single-file output and break a consumer who has no `lib/` next to the bundle.

The declared order in `[lib].modules` is the dependency order:

```
error.cyr  →  png_chunks.cyr  →  png_filter.cyr  →  png_color.cyr  →  png.cyr
```

- `error.cyr` is dependency-free (the `ChitraErr` record + `ChitraErrCode`
  constants).
- `png_chunks.cyr` (byte-buffer cursor + signature/chunk predicates +
  `ChitraPngRaw`) and `png_filter.cyr` (the five spec § 9 unfilter predictors +
  the `chitra_png_parse_raw` framing driver) build on `error.cyr` and on
  `sankoch` (DEFLATE inflate + CRC — *not* chitra's job; see
  [`../../src/lib.cyr`](../../src/lib.cyr) and [`0001-...`](../adr/0001-fork-kii-png-decoder.md)).
- `png_color.cyr` (the canonical-RGBA8 + tRNS color pass) consumes a
  `ChitraPngRaw`.
- `png.cyr` (the public `chitra_png_decode*` API + `ChitraImage`) sits atop all
  three.

Because each module only references symbols defined in a module *earlier* in the
list, the naive top-to-bottom concatenation produces a file where every symbol is
declared before it is used — i.e. it compiles. The stdlib symbols those modules
reference resolve because the consumer's own build environment supplies the
stdlib (the bundle declares its `[deps].stdlib` set in `cyrius.cyml`; resolved by
`cyrius deps` into a real `lib/` dir, which
[`001-lib-must-not-be-symlink.md`](001-lib-must-not-be-symlink.md) covers).

The generated header records the contract in-band:

```
# chitra.cyr -- bundled distribution
# Version: 0.2.1
# Generated by: cyrius distlib
# Do not edit -- rebuild with: cyrius distlib
```

`dist/chitra.cyr` is a build artifact. Edit the `src/*.cyr` modules, then
regenerate — never hand-patch the bundle.

## Consequences and footguns

- **Adding a stdlib `include` to a domain module breaks the bundle.** It will
  compile fine locally (because `lib.cyr` already includes everything and the
  smoke build goes through `lib.cyr`), so the breakage is invisible until a
  consumer links the regenerated `dist/chitra.cyr` without a matching `lib/`
  layout — or until the stripped/duplicated include collides. New stdlib needs go
  in `src/lib.cyr` **and** in `[deps].stdlib` of `cyrius.cyml`, nowhere else.

- **Reordering `[lib].modules` without regenerating + recompiling breaks it.**
  The order *is* the dependency topology. If a future module is inserted out of
  order (e.g. a color helper placed before the chunk parser it depends on),
  concatenation puts a use before its declaration and the bundle stops compiling.
  Any edit to `[lib].modules` must be followed by `cyrius distlib` and a
  recompile of the bundle (the `make test-all` target — version-check + dist
  regen + CPU tests — exercises exactly this path).

- **The two orderings must stay in sync.** `[lib].modules` in `cyrius.cyml` and
  the `include "src/..."` tail of `src/lib.cyr` list the same five files in the
  same order. `distlib` reads the former; the smoke build reads the latter. If
  they drift, the smoke build can pass while the bundle is wrong (or vice versa).
  Treat them as one fact expressed twice.

- **What it affects: every consumer that links `dist/chitra.cyr`.** mabda
  (`gpu_texture_load_png`, a plain `[deps.chitra]` dep, no C shim) and kii (which
  re-folded onto the bundle in its v1.2.0, deleting its own decoder; see its ADR
  0006). A broken bundle is a broken downstream build, not a broken chitra test
  run — the in-tree suites (`make test`) compile through `src/lib.cyr`, so they
  will not catch a bundle-only regression. The `make dist` + smoke-link path is
  the guard for that.
