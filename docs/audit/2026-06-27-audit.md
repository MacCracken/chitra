# chitra — 2026-06-27 Security Audit (JPEG baseline decoder; from-scratch surface)

> **Status**: Second security audit of chitra, gating the 0.3.0 cut. The
> [2026-06-26 PNG audit](2026-06-26-audit.md) verified the PNG decoder's
> fork-parity with kii. This audit covers the **new, from-scratch JPEG
> baseline decoder** added in 0.3.0 — the entropy (Huffman/bit-reader),
> integer IDCT, marker/segment parser, and box-upsample + BT.601 color
> pass. Unlike the PNG decoder, this is **not a fork**: it is original
> Cyrius written for chitra, so it is audited as new code against the
> JPEG-decoder CVE corpus (libjpeg / libjpeg-turbo / stb_image / Go
> `image/jpeg` / Rust `jpeg-decoder`), not for parity.
> **Date**: 2026-06-27 · **Version**: 0.3.0 (pre-tag; `chitra_version()`
> → 300) · **Method**: line-by-line source read of `src/jpeg*.cyr`
> against a 12-class JPEG CVE corpus, cross-checked against the shared
> byte cursor in `src/png_chunks.cyr` and the error enum in
> `src/error.cyr`.

## 1. Scope and methodology

chitra is a pure-Cyrius CPU raster decode **library** — untrusted
encoded bytes in, canonical RGBA8 `ChitraImage` out. No GPU, no C shim,
no external libraries, no CLI, and no terminal/ANSI emit surface (the
sole stderr writer, `chitra_err_print_name`, prints a fixed
decoder-controlled name; see the [PNG audit §2 #8](2026-06-26-audit.md)).
The library threat model — untrusted `(src, len)` span → integer
overflow → undersized alloc → heap overflow — is inherited from the PNG
audit and applies identically to the JPEG path.

The 0.3.0 scope is **JFIF baseline only** (SOF0): sequential Huffman,
8-bit precision, grayscale (1 component) or YCbCr (3 components), chroma
subsampling 4:4:4 / 4:2:2 / 4:2:0 and general `Hi,Vi` with box
upsampling, plus DRI/RST0–7 restart markers. The decode pipeline is:

```
chitra_jpeg_scan_markers   (SOI..SOS marker walk: DQT/DHT/SOF0/DRI parse,
                            reject every non-baseline mode)
  -> _jpeg_parse_sos
  -> _jpeg_decode_scan      (per-component MCU loop:
        _jpeg_decode_block  [bit-reader + DECODE + RECEIVE/EXTEND,
                             DC diff + AC run/size]
        _jpeg_idct_block    [dequant + zig-zag + libjpeg islow integer
                             IDCT + level-shift + clamp]
        plane placement)
  -> box upsample + BT.601 YCbCr->RGB
  -> ChitraImage
```

Public entry points audited: `chitra_jpeg_decode`,
`chitra_jpeg_decode_rgba8`, `chitra_image_decode` (the PNG/JPEG
signature router), and `chitra_jpeg_check_signature`. The decode model
and its deliberate exclusions are recorded in ADR
[`0004-jpeg-decode-model.md`](../adr/0004-jpeg-decode-model.md).

The JPEG decoder spans four new modules:
`src/jpeg_huffman.cyr` (282 lines), `src/jpeg_idct.cyr` (222),
`src/jpeg_markers.cyr` (508), `src/jpeg.cyr` (420). The PNG modules
(`error.cyr`, `png_chunks.cyr`, `png_filter.cyr`, `png_color.cyr`,
`png.cyr`) are unchanged; the JPEG path **reuses** the PNG byte cursor
(`_cur_can_read` / `_cur_u8` / `_cur_u16_be`, `png_chunks.cyr:138-179`)
and the security ceilings defined in `png_chunks.cyr`. JPEG error codes
extend the `error.cyr` enum at 13–23.

Method: each of the 12 JPEG CVE classes in the corpus (§2) was mapped to
the chitra code path that would be exposed if the guard were absent, then
the source was read line-by-line to confirm the guard is present and
behaves as the remediation requires. Every numeric ceiling cited is a
named constant: `CHITRA_MAX_PIXELS` 16777216, `CHITRA_MAX_RAW_BYTES`
268435456 (256 MB), `CHITRA_MAX_DIM` 65535 (`png_chunks.cyr`);
`CHITRA_MAX_COMPONENTS` 4, `CHITRA_MAX_SAMP_FACTOR` 4,
`CHITRA_MAX_BLOCKS_PER_MCU` 10, `CHITRA_MAX_QUANT_TABLES` 4,
`CHITRA_MAX_HUFF_TABLES` 4 (`jpeg_markers.cyr`).

A note on the substrate: Cyrius `>>` is a **logical** shift, so the IDCT
and color passes deliberately use a signed round-to-nearest division
(`_jpeg_descale`, `jpeg_idct.cyr:115-120`) instead of an arithmetic
right shift on signed intermediates. Output was verified
**byte-identical to ImageMagick** on a real 16×16 baseline gradient JPEG
carrying genuine Annex K Huffman tables and AC entropy.

## 2. Convergent hardening checklist mapped to chitra's code

The 12 JPEG CVE classes researched, each mapped to chitra's source.
**PRESENT** = the guard exists and does what the remediation requires.
**N/A — baseline rejects it** = the entire attack surface is eliminated
by chitra rejecting the non-baseline mode at the marker classifier, so
no exploitable code exists to reach. **GAP** = a real missing item.

| # | CVE class (corpus) | Sev | chitra status — evidence (file:line) |
|---|---|---|---|
| 1 | SOF sampling-factor div-by-zero `Hi/Vi=0` (CVE-2018-11212) | high | **PRESENT** — `_jpeg_parse_sof0` rejects `Hi<1`/`Vi<1` (and `>4`) before storing: `jpeg_markers.cyr:348-351` → `CHITRA_ERR_JPEG_SOF`; divisors seeded to 1 at `:334-335` and only grow, so upsample divides (`jpeg.cyr:324,348-350`) can never be 0. No float division anywhere. |
| 2 | DQT/DHT table overruns (bad `Tq`/`Th`, oversized BITS, Σcounts>256, under-read) (stb CVE-2022-28042, GHSL-2023-145/151) | high | **PRESENT** — DQT: `Pq!=0` reject, `Tq>=4` reject, `remaining<64` reject, segment `_cur_can_read`-gated (`jpeg_markers.cyr:182-196`). DHT: `Tc>1`, `Th>=4`, header `<17`, Σcounts`>256`, `remaining<total` all rejected (`:238-263`); symbols write into a 256-entry HUFFVAL bounded by validated `total`. |
| 3 | Huffman over-subscription / incomplete-table OOB symbol index (stb #312, GHSL-2023 series) | high | **PRESENT** — build rejects over-subscription `code >= (1<<L)` → `CHITRA_ERR_JPEG_DHT` (`jpeg_huffman.cyr:72`). | 4 | Entropy index overrun (k>63), bad DC size category (Go #10388, Rust jpeg-decoder #132) | high | **PRESENT** — DC magnitude category `t>16` rejected (`jpeg_huffman.cyr:246`); AC index guarded `k>63` after adding the run, before the store (`:271`) → `CHITRA_ERR_JPEG_ENTROPY`; loop head `k>63` terminates (`:258`); EXTEND/RECEIVE use i64 (no Rust-style unsigned-subtract wrap). |

With dim≤65535, `Hi≤4` ⇒ products stay far inside i64 (no wrap). |
| 8 | DoS via huge dims / excess components / memory exhaustion (Go #10532, #5050) | high | **PRESENT (bounded)** — hard ceilings reject pre-alloc: 16M pixels, 65535 dim, 256 MB raw, `ncomp∈{1,3}` (`jpeg_markers.cyr:318`), ΣHi·Vi≤10 (`:379`). JPEG entropy has no zlib-style amplification. **No decode-time wall-clock/iteration budget** (see §3), but every loop is bounded by validated dims/segment lengths. |
| 9 | Restart (DRI/RSTn) abuse: malformed/missing marker, desync, infinite loop | medium | **PRESENT** — DRI length must be exactly 4 (`jpeg_markers.cyr:295`); `_jpeg_br_restart` requires byte-alignment + `FF D0..D7` and bounds `pos+1<end`, else −1 → `CHITRA_ERR_JPEG_ENTROPY` (`jpeg_huffman.cyr:123-136`, `jpeg.cyr:237`); restart only on `mi%ri==0` with `ri>0` (`jpeg.cyr:233-235`), DC predictors reset (`:240`); MCU loop bounded by `mcu_count` (`:231`). |
| 10 | Color-conversion / chroma-upsample OOB (CVE-2023-2804, CVE-2020-13790, CVE-2021-46822) | high | **PRESENT (reduced surface)** — box (nearest) upsample only via integer `x*hi/max_h`, `y*vi/max_v`; plane dims ≥ image dims so reads are provably in-plane (`jpeg.cyr:324,348-350`). | 11 | Byte-stuffing / marker-scan parse (FF FF fill, FF00 stuffing, truncated segments, missing SOI/SOS) (GHSL-2023, Go truncation) | high | **PRESENT** — SOI check (`jpeg_markers.cyr:116-121`); marker walk gates every step `_cur_can_read(2)` + non-FF reject (`:443-446`), bounded fill-run collapse that consumes ≥1 byte/pass (`:452-458`); APPn/COM skip validates `seglen>=2` + `can_read(seglen-2)` (`:489-497`); bit-reader handles FF00 unstuffing + FFxx marker-stop with zero-pad (`jpeg_huffman.cyr:142-170`); SOS-without-SOF and EOI-before-SOS rejected (`:484,148`). |
| 12 | Shared byte-cursor: every read bounds-checked before access | high | **PRESENT** — JPEG reuses the PNG self-validating cursor: `_cur_can_read` (`png_chunks.cyr:138-142`, rejects `n<0` and `remaining<n`), enforced by `_cur_u8` (`:147-153`) and `_cur_u16_be` (`:172-179`), each re-checking and returning 0 on OOB. |

| — | **Benchmark / decode-budget harness (DoS time-bound regression)** | medium | **GAP** — no benchmark harness in-tree; decode latency unmeasured (v1.0 gate). See §3. |
| — | **Differential corpus vs a reference decoder** | medium | **PRESENT (partial)** — byte-identical vs ImageMagick on one 16×16 baseline gradient; 199 `jpeg.tcyr` assertions. A broader differential corpus is a gap. See §3/§5. |

**Checklist verdict**: of the 12 CVE classes, **ten are PRESENT** (live
guards at the cited lines — classes 1–4 and 7–12) and **two are N/A**
because the baseline scope rejects the entire surface at the marker
classifier (class 5 progressive; class 6 arithmetic / extended / lossless
/ 12-bit). Within class 10, the CMYK/PPM/BMP sub-surfaces are likewise
unreachable (no 4-component or non-JPEG path), and class 3 carries one
bounded-DoS caveat (no decode-time budget). The only true checklist
**GAPs** are the missing in-tree fuzz and benchmark harnesses and the
narrow differential corpus — all testing-posture, all on the v1.0 roadmap.
No memory-safety guard is missing.

## 3. Findings by severity

The line-by-line read confirmed **all 30+22+8+21 = 81 claimed guards
across the four modules (jpeg_markers / jpeg_huffman / jpeg_idct / jpeg)
are genuinely present and behave as specified**.
The marker parser, the entropy/bit-reader, the integer IDCT, and the
scan/upsample/color path are **bounds-tight on the untrusted byte and
entropy surface**: no reachable out-of-bounds read/write, no integer
overflow into an undersized allocation, and no division-by-zero was
found. Soundness and CVE-corpus parity are therefore **complete for the
baseline scope**.

There are **no CRITICAL, HIGH, or MEDIUM code findings.** The findings
below are one MEDIUM **process** finding (no fuzzing of this surface) and
a set of LOW items that are leniency/robustness choices or unchecked-
allocation hygiene, none of which is an exploitable memory-safety defect.

### CRITICAL — none
### HIGH — none
### MEDIUM

- **No in-tree fuzz harness over the JPEG entropy / byte-buffer surface**
  (`jpeg.cyr`, `jpeg_huffman.cyr`, `jpeg_markers.cyr`). Confirmed:
  `find . -iname '*fuzz*'` returns nothing; no harness drives
  `chitra_jpeg_decode` / `chitra_image_decode` with adversarial spans.
  Every MCU/restart/entropy bound documented in §2 is a **manual proof
  only**. The decoder's hardening *lineage* is the kii/PNG fork's, but
  the JPEG entropy decoder is from-scratch chitra code and its
  byte-buffer/entropy surface has **never been fuzzed in this tree**.
  This is a testing-coverage gap, not a code defect — but for a
  from-scratch entropy decoder it is the single highest-leverage
  hardening investment, hence MEDIUM rather than LOW. v1.0 gate.

### LOW

- **Truncated / early-EOI entropy stream is silently zero-padded, not
  reported as truncated.** When the entropy data ends before all MCUs are
  decoded and `restart_interval==0`, `_jpeg_br_load_byte` synthesizes an
  EOI / leaves `cnt==0` past the end (`jpeg_huffman.cyr:149,153-156`) and
  `_jpeg_br_bit` returns 0 forever (`:179`). `_jpeg_decode_block` has no
  end-of-data signal, and the MCU loop's final check (`jpeg.cyr:296`)
  only tests `err_out`, which is never set for plain end-of-data. A
  truncated baseline image therefore yields fabricated (mostly-zero)
  trailing MCUs instead of `CHITRA_ERR_JPEG_ENTROPY`. **Fully
  memory-safe** (all reads/writes remain bounded) — a leniency choice
  matching stb_image, weaker than the fail-fast PNG sibling. Consumers
  cannot distinguish a complete decode from a truncated one.

- **DC magnitude category accepted up to 16 (baseline spec max 11); AC
  size up to 15.** `_jpeg_decode_block` rejects only `t>16`
  (`jpeg_huffman.cyr:246`); AC `s = rs & 15` (`:264`) has no `s<=10`
  bound. `_jpeg_br_bits` then reads up to 16/15 bits into an i64 —
  **memory-safe** (the `t>16` cap prevents `1<<t` overflow and the index
  guards prevent OOB), but an out-of-spec stream decodes instead of being
  rejected as `CHITRA_ERR_JPEG_ENTROPY`. Spec laxity, not a hole.

- **Duplicate SOS component selector (`Cs`) not rejected.** `jpeg.cyr:89-96`
  sets `td/ta` on the matching frame component but does not detect a `Cs`
  repeated across SOS components (contrast SOF0's duplicate-id check at
  `jpeg_markers.cyr:356-363`). A scan listing the same `Cs` twice
  overwrites one component's selectors and leaves another at its
  zeroed-alloc default (table 0). Because the allocator returns
  kernel-zeroed pages, the worst case is decoding with the wrong/absent
  table — caught downstream by the huff-present check (`jpeg.cyr:162/164`,
  `CHITRA_ERR_JPEG_SOS`) or a corrupt image. **Memory-safe**; pure spec
  laxity.

- **`_jpeg_zigzag_table` does not check `alloc()` for failure.**
  `jpeg_idct.cyr:44` (`var t = alloc(64 * 8)`) is followed immediately by
  64 `store64(t + ...)` with no `if (t == 0)` guard — inconsistent with
  every other JPEG allocation (`jpeg.cyr:181-182,202-206,299-301`, all of
  which check). If the allocator returned 0 here, 512 bytes would be
  written from address 0. **Reachability is genuinely low**: this is a
  one-time 512-byte lazy singleton (`_jpeg_zz_ptr`, `jpeg_idct.cyr:43`)
  built only *after* far larger checked allocations (planes, `zz`/`out`)
  have already succeeded, so OOM is normally caught earlier. Real missing
  guard, low severity; recommend adding the `if (t == 0)` check for
  consistency.

- **No in-tree benchmark / decode-budget harness.** No benchmark file in
  tree; decode latency and worst-case decode time are unmeasured. All
  loops are bounded by validated dims/segment lengths, so this is a
  DoS-*latency* observability gap, not an unbounded-loop defect. v1.0
  gate.

### Resolved in the 0.3.0 cut

Two of the LOW spec-laxity findings were **fixed in this release**, each
with a regression test:

- **DC/AC category bounds tightened** — `_jpeg_decode_block`
  (`jpeg_huffman.cyr`) now rejects DC category `t>11` and AC size `s>10`
  (8-bit baseline maxima, T.81 Tables F.1/F.2) as `CHITRA_ERR_JPEG_ENTROPY`.
- **Duplicate SOS component selector rejected** — `_jpeg_parse_sos`
  (`jpeg.cyr`) now rejects a repeated `Cs` as `CHITRA_ERR_JPEG_SOS`.

The §4 doc-drift comments were corrected in the same cut. **Deferred to
v1.0** (documented, not blocking, all memory-safe): truncation / early-EOI
detection (currently zero-padded), the unchecked one-time
`_jpeg_zigzag_table` alloc, and the in-tree fuzz + benchmark harnesses.

## 4. Doc-drift (stale comments — found and corrected in the 0.3.0 cut)

Comment-only; none changed a code path or affected safety. All corrected
in this release:

- **`jpeg_markers.cyr` `_jpeg_marker_action` header** stated action values
  are "1..6" but the classifier returns 7 for DRI (`0xDD`) — corrected to
  the 1..7 convention (matching architecture note 004).
- **`jpeg_markers.cyr:17` (module header) and `:151` (action-4 comment)**
  had described DRI as *skipped*, but `0xDD` returns action 7
  (`jpeg_markers.cyr:134`) and `_jpeg_parse_dri` fully parses `Ri` into
  `JF_RESTART_INTERVAL` — both comments now say DRI sets the restart interval.
- **`jpeg_markers.cyr` frame-layout prose** had listed stale offsets
  (`+80` components, `+208`/`+216` huff) diverging from the authoritative
  vars — now corrected to `JF_SOS_OFFSET=80`, `JF_HUFF=88`,
  `JF_HUFF_PRESENT=96`, `JF_COMP=104` (stride 48).
- **`jpeg_idct.cyr`** had said *"the two arrays zz/quant/out"* (it names
  three) — now reads "the arrays".

This audit is the record that these comments were stale at audit time; all
were corrected in the 0.3.0 cut.

## 5. Residual risk and recommendations

- **The from-scratch entropy decoder has never been fuzzed in-tree
  (principal residual risk).** Every bound in §2 is a manual proof. The
  highest-leverage investment is an **in-tree fuzz harness** targeting the
  JPEG-original surface specifically: random `(src, len)` spans into
  `chitra_jpeg_decode`; crafted DQT/DHT segments exercising the table-id /
  Σcounts / over-subscription rejects (`jpeg_markers.cyr:182-263`,
  `jpeg_huffman.cyr:72`); crafted SOF0 sampling factors hammering the
  CVE-2018-11212 guard (`:348-351`); adversarial entropy streams driving
  the `k>63` / `t>16` / EXTEND paths (`jpeg_huffman.cyr:246,258,271`); and
  truncated/early-EOI streams to characterize the zero-pad leniency (§3).
  This reproduces in-tree the fuzz property the PNG decoder inherited from
  its kii lineage but which the JPEG decoder never had.

- **Differential fuzzing vs a reference decoder.** The byte-identical
  result vs ImageMagick is currently a single 16×16 gradient. Extend it
  to a differential-fuzz loop (chitra vs libjpeg/ImageMagick over a
  generated corpus: every subsampling ratio, grayscale + YCbCr, with/
  without DRI, edge dimensions near the 65535 / 16M caps). This catches
  *correctness* divergences the unit suite cannot, and surfaces any
  decode that should have errored but didn't (the truncation-leniency
  class).

- **Benchmark / DoS-latency matrix.** Add an in-tree benchmark harness and
  record a first decode-latency matrix across subsampling ratios and the
  dimension ceilings, then decide whether a decode-time/iteration budget
  is warranted on top of the existing size caps (§2 class 8). Currently
  unmeasured.

- **Tighten the spec-laxity LOW items** (optional, defense-in-depth):
  reject DC category >11 / AC size >10 as `CHITRA_ERR_JPEG_ENTROPY`;
  reject duplicate SOS `Cs`; add the missing `alloc` null-check at
  `jpeg_idct.cyr:44`; and decide whether mid-scan truncation should be a
  hard `CHITRA_ERR_JPEG_ENTROPY` rather than a zero-pad. None is required
  for memory safety.

- **Re-audit on feature growth.** Adding progressive JPEG, arithmetic
  coding, 12-bit precision, CMYK/YCCK, or any new emit/output surface
  re-opens the CVE classes this audit marked **N/A — baseline rejects it**
  (§2 classes 5, 6, and the CMYK/writer sub-surfaces of 10). The
  baseline-only posture is the load-bearing simplification; see ADR
  [`0004-jpeg-decode-model.md`](../adr/0004-jpeg-decode-model.md).

The fuzz and benchmark harnesses are tracked as v1.0 gates in the
[roadmap](../development/roadmap.md); this audit is the JPEG-specific
record behind those gates.
