# chitra — 2026-06-26 Security Audit (first audit; fork-parity verification)

> **Status**: First security audit of chitra. Versions 0.1.0 → 0.2.1
> shipped without one — this closes that gap. chitra is a *fork* of
> kii's W3C-compliant PNG decoder, so this audit's job is **fork-parity
> verification**: prove chitra's actual source carries every guard from
> kii's convergent hardening checklist, then audit chitra's *new* surface
> (byte-buffer I/O boundary, canonical-RGBA8 + tRNS color pass, sub-byte /
> 16-bit / Adam7) that did not exist in the fork point.
> **Date**: 2026-06-26 · **Version**: 0.2.1 · **Method**: line-by-line
> source read of `src/*.cyr` against the upstream checklist.

## 1. Scope and methodology

chitra is a pure-Cyrius CPU raster decode **library** — it turns
untrusted encoded image bytes into canonical RGBA8, zero GPU, no C shim,
no external binaries. Critically for this audit's threat model: chitra
has **no CLI and no terminal/ANSI output**, and the only byte sink it
touches is `chitra_err_print_name` ([`../../src/error.cyr`](../../src/error.cyr)),
which writes a *fixed, decoder-controlled* error name to stderr (never
attacker-derived bytes). It is linked by consumers (mabda's `gpu_texture_load_png`,
and kii itself since its v1.2.0 re-fold) who hand in in-memory bytes and
receive a pixel buffer. The entire kii §3.4 terminal-escape attack class
(the dominant RCE surface in the upstream audit) is therefore **N/A for
chitra by construction** — there is no byte sink that reaches a terminal.

chitra was forked **one-time** from kii's `src/png.cyr` (proven,
fuzz-hardened, W3C-compliant) and carries **no live dependency** on it —
bugfixes cross the boundary as manual backports. The upstream CVE
research is therefore not re-derived here. The authoritative 140-row
cross-substrate CVE corpus (libpng / lodepng+stb / zlib / terminal
emulators) lives in kii's audit at
[kii's 2026-05-22 audit](https://github.com/MacCracken/kii/blob/main/docs/audit/2026-05-22-audit.md)
and is **referenced, not duplicated**. This audit verifies that the
hardening that corpus produced is present in chitra's bytes, and extends
coverage to the surface chitra added on top of the fork:

1. **The byte-buffer I/O boundary** — kii read from a file path; chitra
   reads from an untrusted `(src, len)` in-memory span. Every read must
   be bounds-checked against `len` *before* the access. This is new code
   (`src/png_chunks.cyr` cursor) and is audited fresh.
2. **The canonical-RGBA8 normalization + tRNS resolution color pass**
   (`src/png_color.cyr`) — new code: 16-bit→high-byte, sub-byte
   grayscale ×255/85/17 scaling, palette index → PLTE lookup, tRNS
   alpha resolution.
3. **Feature widening** — Adam7 interlace (7-pass deinterlace) and the
   full spec-legal bit-depth set (1/2/4/8/16 per color type), neither of
   which the fork point supported. These are audited as new bounded paths.

DEFLATE is **not chitra's code**: IDAT inflate, CRC-32, and adler32 are
sankoch's job (RFC 1950/1951). sankoch-internal hazards are upstream
items (§5).

The substrate threat model — integer-overflow → undersized alloc →
heap-overflow as the dominant single-file-decoder failure chain — is
inherited verbatim from the upstream corpus.

## 2. Convergent hardening checklist mapped to chitra's code

The upstream checklist's ten patterns, each mapped to chitra's actual
source and verified line-by-line. "PRESENT" means the guard exists and
does what the upstream remediation specified.

| # | Pattern | chitra status | Evidence (file:line) |
|---|---|---|---|
| **1** | IHDR width×height×bpp overflow / dimension cap | **PRESENT** | `png_filter.cyr:191-197` — `width/height <= 0`, `> CHITRA_MAX_DIM`, `width*height > CHITRA_MAX_PIXELS`; per-side cap (65535) executes *before* the multiply so the product ≤ ~4.29e9 cannot wrap i64. Constants `png_chunks.cyr:37-39`. Re-asserted in the color pass `png_color.cyr:80-88`. Post-IHDR derived sizes capped `png_filter.cyr:500-507`. |
| **2** | IDAT chunk-fusing accumulator cap | **PRESENT** | `png_filter.cyr:437-441` — `idat_total += chunk_len; if (idat_total > CHITRA_MAX_RAW_BYTES) return CHITRA_ERR_OOM`. |
| **3** | Decompression-bomb / ratio cap (DEFLATE 1032:1 ceiling) | **PRESENT** | `png_filter.cyr:519` — `(inflated_size / idat_total) > CHITRA_MAX_INFLATE_RATIO` (=1100, `png_chunks.cyr:43`); divisor proven non-zero by the no-IDAT reject at `:512`. |
| **4** | §11.2.2 Table 11.1 color-type × bit-depth allow-list | **PRESENT (stronger than fork point)** | `png_filter.cyr:175-185` — per-ct depth table (ct0: 1/2/4/8/16; ct3: 1/2/4/8; ct2/4/6: 8/16); `depth_ok == 0 → CHITRA_ERR_BIT_DEPTH`. ct3+depth16 rejected. Color-type allow-list `png_chunks.cyr:210-216` (−1 for unknown) → `png_filter.cyr:164-165`. |
| **5** | Palette index OOB on scanline | **PRESENT (hard reject, not substitute)** | `png_color.cyr:292-295` (depth-8) and `:173` (sub-byte) — `idx >= entry_count → CHITRA_ERR_BAD_CHUNK`. PLTE span re-validated within `(src,len)` at `:277-280` / `:158-161`; non-empty PLTE required `:272-275` / `:157`. Note: chitra **rejects** OOB indices rather than substituting black (the fork's behaviour) — a stricter posture. |
| **6** | Per-chunk length cap + IEND zero-length | **PRESENT** | `png_filter.cyr:407` per-chunk `chunk_len > CHITRA_MAX_RAW_BYTES → BAD_CHUNK`; span fits remaining before any CRC scan `:412`; IEND `chunk_len != 0 → BAD_CHUNK` `:465`. |
| **7** | Chunk-ordering enforcement | **PRESENT** | First chunk forced through IHDR parse after the 8-byte signature skip (`png_filter.cyr:380-383`, `:138`); duplicate PLTE / PLTE-after-IDAT / `>768` / `%3!=0` all `BAD_CHUNK` `:447-450`. |
| **8** | Stderr path-echo ANSI-injection | **N/A (library)** | chitra has no terminal surface and emits no attacker-derived bytes. Its only stderr writer, `chitra_err_print_name` (`error.cyr:84`), prints a *fixed, decoder-controlled* error name (sourced from `chitra_err_name`, `error.cyr:65-80`) — never user/path bytes — so the path-echo ANSI-injection vector kii closes has no analog here. The upstream finding does not apply. |
| **9** | Short-read silent continuation / fail-fast | **PRESENT** | Self-validating cursor rejects every over-read: `_cur_can_read` (`png_chunks.cyr:138-142`, `n<0` and `remaining<n`), enforced by `_cur_u8` `:148` and `_cur_u32_be` `:159`. Walk-level truncation `png_filter.cyr:396-401` (rem==0 stops, rem<8 → TRUNCATED) and `:412`. Inflate output pinned to the declared budget `:554-556` (`n<0` and `n!=inflated_size` both → INFLATE). |
| **10** | Zero-progress inflate starvation | **UPSTREAM (sankoch)** | DEFLATE is sankoch's job; chitra cannot see the inflate loop. chitra's pre-inflate caps (#1/#2/#3) bound the input handed to `zlib_decompress` and the exact-size check (`:556`) catches a malformed output. The zero-progress two-cycle invariant itself is a sankoch audit item (§5). |

**Parity verdict**: every upstream pattern that applies to a library is
PRESENT in chitra's source at the cited lines; #8 is genuinely N/A
(no emit surface); #10 is upstream-by-design. The fork carried the
hardening across intact, and item #4 is actually *stronger* in chitra
(full Table-11.1 allow-list vs the fork's tier-1 8/16-only subset).

## 3. chitra-specific findings

The full guard inventory verified during this audit (drawn from the
line-by-line read of all five modules) is **complete**: 27 guards in
`png_chunks.cyr`, 35 in `png_filter.cyr`, 21 in `png_color.cyr`, and
the decode-orchestration / error guards in `png.cyr` + `error.cyr` are
all genuinely present and correctly behave as claimed. **No CRITICAL,
HIGH, or MEDIUM code finding was found.** The integer-overflow chain
that defines the upstream threat model is closed: every `width*height`
multiply is preceded by a per-side cap (`png_filter.cyr:192`,
`png_color.cyr:81-84`), and every scanline read is gated by the
`scan_len >= height * row_bytes` sufficiency check (`png_color.cyr:96`).

### 3.1 New-surface audit (byte-buffer boundary, color pass, Adam7/sub-byte)

The new code that did not exist at the fork point was the highest-risk
area to verify. All of it is bounded:

- **Byte-buffer cursor** — the new I/O boundary. Self-validating: no
  `load8` happens without a preceding `_cur_can_read` (`png_chunks.cyr:148,159`),
  and `_cur_u32_be` masks to 32 bits (`:166`) so a high-bit chunk length
  can never sign-extend into a negative i64.
- **Adam7 deinterlace** (`png_filter.cyr:269-345`) — the inflated size is
  recomputed for the 7-pass layout (`_chitra_adam7_inflated_size`,
  `png_filter.cyr:247-261`, used at `:496`), capped (`:500`), then pinned
  to the actual inflate output (`:556`). Pass buffers are sized
  `row_bytes_full` (`:282-284`); per-pass `prow_bytes <= row_bytes_full`
  since `pass_w <= width`; inflated reads advance by `1 + prow_bytes`
  (`:309`) summing exactly to `inflated_size`; scatter targets are bounded
  `full_y < height` / `full_x < width` (`:315,320,331`) so the destination
  stays inside `height * row_bytes_full == pixel_size`. CVE-2014-9495 /
  16-bit-interlaced / stb-shift-UB classes are bounded here, not unhandled.
- **Sub-byte unpack/pack** — `shift = 8 - depth - bit_off`
  (`png_chunks.cyr:240`) cannot go negative for depth ∈ {1,2,4} with
  `bit_off` a multiple of depth within 0..7 (min shift 0); reads/writes
  are masked (`:243,255-257`). The stb `shiftsigned` UB class does not
  reach chitra.
- **tRNS color pass** — every tRNS span is re-validated within `(src,len)`
  before any deref (`png_color.cyr:109-113`), with per-color-type length
  checks (`!=2` gray `:128,:200`; `!=6` RGB `:234`; palette
  `trns_len > entry_count` `:163,:284`), and per-entry tRNS reads are
  bounded by `idx < trns_len` (`:176,:298`).

### 3.2 Process / LOW findings

With parity complete and the new surface bounded, the only findings are
process-level:

| Severity | Finding | Detail |
|---|---|---|
| LOW | **No in-tree fuzz harness** | The README describes the decoder as "fuzz-corpus-tested" — that is true of its **kii lineage**, but chitra itself ships **no fuzz harness file** in-tree. The 525 assertions across `tests/tcyr/` are deterministic unit/integration suites, not differential fuzzing. The byte-buffer boundary and the new color/Adam7 paths deserve their own corpus. |
| LOW | **No in-tree benchmark harness** | Despite linking the stdlib `bench` module, there is no benchmark file in-tree; decode latency is **not yet measured**. |
| LOW | **Doc-drift in `error.cyr` / source headers** | Stale comments that no longer describe runtime behaviour. See §4. Cosmetic; zero memory-safety impact. |

## 4. Doc-drift (stale comments — cosmetic, no behavioral gap)

These are real and should be corrected to keep the source honest, but
none changes a code path or affects safety. All were verified against the
live code:

- **`error.cyr:26`** — `CHITRA_ERR_INTERLACE` comment reads
  *"interlace != 0 (Adam7) — single-pass only (chitra 0.2)"*, but
  `png_filter.cyr:171` now accepts `interlace == 1` and the full 7-pass
  deinterlace runs (`png_filter.cyr:269-345`). The code now only fires for
  an **invalid interlace method byte** (not 0 or 1).
- **`error.cyr:27`** — `CHITRA_ERR_BIT_DEPTH` comment reads
  *"bit_depth != 8 — 1/2/4/16 are chitra 0.2 / AL.P0d scope"*, but
  `png_filter.cyr:175-185` accepts the full spec depth set per color type.
  The code now fires only for an **illegal color_type × bit_depth combo**
  (e.g. ct3+depth16), not `bit_depth != 8`.
- **`error.cyr:74-75`** — the error-name strings `"interlace unsupported"`
  and `"bit depth unsupported"` misreport now-supported features; they
  should read as "invalid interlace method" / "illegal depth×colortype".
  Library-internal strings only — no ANSI/path-echo surface (see §2 #8).
- **`png_color.cyr` header (≈ lines 19-22, 66-70)** — claims *"only bit
  depth 8 … no bit unpacking"* and *"sub-byte depths never reach here"*,
  contradicted by the live 16-bit `bps = bit_depth/8` stride and the
  sub-byte block (`png_color.cyr:121-191`).
- **`png_chunks.cyr:9-11,31`** — states the security core / caps are
  *"ported verbatim from kii"*. No longer literally accurate after the
  Table-11.1 widening and the Adam7 / sub-byte additions, which are
  chitra-original. The *caps themselves* match kii; the surrounding
  decode is no longer a verbatim port.

## 5. Residual risk and upstream

- **sankoch-internal DEFLATE hazards** are the principal residual risk
  and are **out of chitra's code**. The three spec-mechanic zlib classes
  that transfer to any DEFLATE impl (Huffman-table construction /
  code-length completeness / zero-progress inflate — kii's checklist #10,
  upstream CVE-2004-0797 / 2005-1849 / 2005-2096 classes) live in sankoch.
  chitra's mitigation surface is the **pre-inflate caps** (#1/#2/#3): it
  bounds the IDAT input and the requested output, then enforces an
  exact-size match on the result (`png_filter.cyr:554-556`). If sankoch
  ships a Huffman bug, chitra catches the *symptom* (output-size mismatch
  → `CHITRA_ERR_INFLATE`) but cannot fix the cause. Recommend filing /
  tracking these as upstream sankoch audit items, cross-referencing this
  doc.
- **No in-tree fuzz/bench harness yet** (§3.2). The "fuzz-corpus-tested"
  property is inherited from the fork point, not reproduced in this repo.
  The byte-buffer boundary and the color/Adam7 paths are chitra-original
  and have **never been fuzzed in this tree**. This is the single most
  valuable hardening investment available.
- **Bump-allocator, no free** — chitra runs on the stdlib bump allocator;
  `chitra_image_free` / `chitra_raw_free` are no-ops
  (`png_chunks.cyr:103-105`). This eliminates the use-after-free /
  double-free CVE classes that dominate the upstream corpus (those are
  all N/A under bump allocation), at the cost of per-decode arena growth —
  an accepted residual for the texture-load use case. See
  [`../architecture/003-bump-allocator-no-free.md`](../architecture/003-bump-allocator-no-free.md).
- **tRNS for color_type 4/6** is silently ignored rather than rejected
  (`png_color.cyr:310-339` never read `trns_ptr`). This is benign — a
  spec-illegal tRNS-with-alpha-color-type has no memory-safety impact
  since alpha comes from the pixel — but a stricter decoder would reject
  it. Accepted leniency, not a gap.

## 6. Recommendations / carried forward

1. **Add an in-tree fuzz harness** (LOW→actionable; highest leverage).
   Target the chitra-original surface specifically: random `(src, len)`
   spans against the byte-buffer cursor; crafted IHDR exercising the
   #1/#2/#3 caps; random palette index vs PLTE length; crafted sub-byte
   and Adam7 layouts; 16-bit stride with adversarial filter bytes. This
   reproduces in-tree the property the README currently borrows from the
   fork lineage.
2. **Add an in-tree benchmark harness** and record a first decode-latency
   matrix (currently *not yet measured*); the `bench` stdlib module is
   already a dependency.
3. **Fix the doc-drift in §4** — one-line comment corrections in
   `error.cyr` (lines 26, 27, 74, 75), `png_color.cyr` header, and
   `png_chunks.cyr:9-11,31`. Durable-vs-volatile: these are source
   comments, fix them in source; this audit is the record that they were
   stale at 0.2.1.
4. **Track the sankoch upstream items** (Huffman-table / code-length /
   zero-progress) as external gates; mark resolved if sankoch already
   implements the bounds.
5. **Re-audit on feature growth.** Adding a new format (JPEG/GIF/BMP),
   ancillary chunk decode, or any new emit/output surface re-opens classes
   this audit marked N/A — in particular, any future stdout/path-echo
   surface re-activates the entire kii §3.4 terminal-escape corpus that is
   currently N/A for chitra.
